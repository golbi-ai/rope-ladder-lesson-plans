Architecture

Godot’s main layer bootstraps the engine and computes frame timing; platform modules supply OS, display, input, storage, and deployment integration. The Android implementation combines C++/JNI adapters with Java/Kotlin hosts, plugin and storage services, and a Gradle-driven exporter/build; iOS and Linux/BSD modules provide parallel platform back ends.

Internal pathways

How responsibilities inside this subsystem collaborate.

  • Core containers → Dynamic value runtime

    Array and Dictionary are dynamic containers implemented alongside the generic container layer.

  • Dynamic value runtime → Deferred and worker execution

    Deferred messages carry Variant argument pointers for calls.

  • Dynamic value runtime → Object runtime

    Objects accept Variant values for dynamic property access, calls, metadata, and signals.

  • Dynamic value runtime → Reflection metadata

    Method and property metadata describe Variant-facing types and arguments.

  • GDExtension runtime → Resource loader

    Extension resource-format support allows GDExtension resources to participate in resource loading.

  • Geometry and transforms → Dynamic value runtime

    The value runtime has pointer-conversion and internal-accessor support for geometry value types.

  • Geometry and transforms → Spatial acceleration

    Spatial-query structures organize and query geometry bounds and triangle data.

  • Native extension host → Reflective object model

    Native extensions register extension classes in ClassDB.

  • Object runtime → Script integration

    Script instances attach to Objects and script language classes derive from Object-facing types.

  • Platform foundation → Deferred and worker execution

    The worker and message systems rely on thread and synchronization primitives.

  • Project settings → Input system

    InputMap has a load_from_project_settings operation for loading action definitions.

  • Project settings → Packed data

    ProjectSettings exposes resource-pack loading.

  • Reflection metadata → Object runtime

    Object classes obtain GDType metadata and ClassDB registers their exposed members.

  • Resource loader → Packed data

    Resource loading resolves paths whose file access can be supplied by packed-data sources.

  • Text and localization → Object runtime

    Objects use translation domains and interned names in their runtime APIs.

  • Virtual-method build generation → Reflection metadata

    Generated virtual-method support participates in the object and class-binding layer.

How responsibilities inside this subsystem collaborate.

  • C# source generators and analyzers → GodotSharp managed runtime

    Generated script metadata and registration methods are consumed by the runtime bridge for many managed script types.

  • Completion fixtures → Scene fixtures

    Many completion configurations may reference one scene fixture as contextual input.

  • FBX-to-GLTF importer → GDScript editor services

    The FBX editor importers are editor-facing import services alongside the GDScript editor services.

  • GDScript front end and compiler → GDScript editor services

    Editor highlighting, documentation, and translation parsing inspect tokenizer and parser structures.

  • GDScript front end and compiler → GDScript language-server protocol

    The language-server parser extension derives document symbols, links, diagnostics, and completions from GDScript parsing.

  • GDScript resource and cache storage → GDScript front end and compiler

    Stored GDScript source and token buffers are inputs to parsing and compilation.

  • Godot .NET editor tools → IDE messaging

    The editor tools host messaging servers, while IDE clients exchange many requests and responses over each connection.

  • GodotSharp managed runtime → C# source generators and analyzers

    The managed runtime exposes partial API and callback types that generators complete at compile time.

  • GodotSharp managed runtime → Godot .NET editor tools

    The editor tools call managed Godot API and internal bridge entry points.

  • GridMap → native 3D physics server

    GridMap obtains PhysicsServer3D body state while processing its octant-backed world content.

  • Jolt 3D physics backend → native 3D physics server

    Both expose a PhysicsServer3D implementation, making them alternative backend implementations of the same engine-facing contract.

  • Managed plugin loader → Godot .NET editor tools

    One plugin load context loads the editor-tools assembly and resolves its dependencies.

  • Mono/.NET native host → GodotSharp managed runtime

    One native host supplies one ordered native interop function table that the managed runtime initializes against.

  • OpenXR action map → GodotSharp managed runtime

    Managed bindings expose the persisted OpenXR action-map resource graph.

  • Procedural and media resources → GodotSharp managed runtime

    Managed API bindings expose resource classes implemented by the native procedural and media modules.

  • Scene multiplayer → 2D navigation server

    Both are runtime services; multiplayer replicates scene state while navigation computes movement data for scene nodes.

  • Scene multiplayer → GodotSharp managed runtime

    Managed RPC attributes and generated method metadata are exposed to the scene multiplayer implementation.

  • Visual shader editor → Visual shader model

    The editor presents and edits the VisualShader graph model.

  • WebRTC networking → Browser platform bridges

    Web-platform WebRTC implementations bridge peer connections and data channels to browser APIs.

  • WebSocket networking → Browser platform bridges

    Web-platform WebSocket support delegates browser socket interaction to JavaScript.

  • glTF document translator → glTF accessor and buffer-view codec

    GLTFDocument uses accessors to decode animation and mesh values and to encode exported data.

  • glTF document translator → glTF skeleton construction

    The document conversion flow delegates skin-node analysis and skeleton construction to SkinTool.
